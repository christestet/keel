# Keel 0.1.0 release readiness

This is a non-normative release-readiness note for the first public Keel
version. The language remains governed by the specification, KDRs, and
conformance suite. The release mechanics are governed by
[`release-process.md`](release-process.md).

## Release label

Keel 0.1.0 should be presented as a **developer preview**, not as production
ready infrastructure. The honest compatibility statement is:

- source language and toolchain behavior are conformance-backed where listed in
  [`feature-status.md`](feature-status.md);
- syntax, diagnostics codes, formatter output, generated Go, and audit ordering
  are treated as compatibility-sensitive;
- package registry, native backend, OCI output, FFI, and production support
  windows are not part of 0.1.0;
- executable generation still depends on the Go toolchain.

A source-only tag can be cut earlier as an internal milestone checkpoint, but it
must not be announced as a usable public toolchain release. A public 0.1.0
developer preview needs an installation story and validation snapshot.

## Minimum public 0.1.0 gate

Before publishing 0.1.0 as a public developer preview, all of the following must
be true:

| Area | Required state |
|---|---|
| M7 conformance | `KEEL_MILESTONE=M7 scripts/preflight.sh` is green with `221 passed, 0 failed, 4 skipped`. |
| Query pipeline | `keel check`, `run`, `test`, and `build` use the M8 query path without observable diagnostic, formatter, generated-source, or runtime drift. |
| Performance | The M8 reference machine is filled in, `baseline.tsv` has nonzero baselines, and the KDR-0019 budgets are enforced in CI. |
| Release artifacts | There is a documented install path for at least macOS and Linux, or the release is explicitly labelled source-only and not positioned as a usable public toolchain. |
| Version reporting | `keel --version` and `keelc --version` report the release version and commit. |
| Documentation | Getting started, language tour, feature status, compatibility, troubleshooting, and examples match the release binary. |
| Examples | A curated set of examples builds/runs with the release compiler and records the exact validation command. |
| Go dependency | Go is documented as a required backend toolchain dependency for 0.1.0. SQL cases document whether `modernc.org/sqlite` is resolved from cache or network. |
| Known limitations | Missing registry, native backend, OCI output, FFI, `keel fix`, and unsupported stdlib surfaces are listed prominently. |

## LSP policy for 0.1.0

Resolved: ship `keel lsp`. All ten `tests/lsp/m8-base` golden transcripts pass
byte-for-byte against the real `keelc-lsp` server (verified in CI via
`cargo test --workspace`), covering every advertised M8 base capability —
incremental sync, diagnostics, go-to-definition, hover, completion, and
document symbols. The remaining caveat for docs/release notes: those four
semantic capabilities resolve module-level `fn`/`struct` declarations by name
only, not local scopes (parameters, `let` bindings) — call this out in
known limitations rather than implying full-fidelity go-to-definition.

Do not ship a server that advertises a capability without a passing golden
transcript for it — that bar is what "ship it" means here, not "the server
exists."

## Explicitly not required for 0.1.0

The following are important but should not block a developer-preview release if
the minimum gate above is met:

- native backend;
- package registry or publishing;
- reproducible OCI image output;
- C FFI;
- OpenAPI client/server generation;
- package lockfiles or dependency version resolution;
- production support/backport window;
- 1.0 compatibility declaration.

## Current blockers

As of the current M8 implementation slice:

- the performance fixture runs in CI (`m8-benchmark` job) on the reference
  machine and has produced a real captured baseline, but all three KDR-0019
  budgets are currently exceeded on that machine (`keel_check` 794 ms vs.
  300 ms, `keel_build_cold` 18.8 s vs. 10 s, `keel_build_incremental` 1.7 s
  vs. 1 s) — this is a real compiler-performance gap, not a measurement
  artifact, and the `--enforce` regression gate stays off until it closes.
  See [`docs/m8-status.md`](m8-status.md) for the numbers and next steps;
- `keel_build_incremental` specifically is expected to stay a documented
  `--known-gap` even after the other two budgets are met: each `keel build`
  invocation creates a fresh Salsa database, so there is no cross-invocation
  incremental reuse yet (only same-process reuse, as `keel lsp` could exploit
  but does not yet);
- `--version` is not implemented (in review: PR #25 adds
  `keel --version` / `keelc --version` with `KEEL_BUILD_COMMIT` embedding);
- release artifacts: `.github/workflows/release.yml` now builds Linux
  x86_64 and macOS arm64 tarballs with checksums on a version tag and
  attaches them to a **draft** GitHub release — publishing the draft (and
  any signing/notarization story) remains a human decision and is still
  unexercised;
- public docs describe the source-build workflow first;
  [`getting-started.md`](getting-started.md) now also documents the
  release-tarball install path that the workflow produces.

## Validation evidence to record

The 0.1.0 release commit must record:

```text
scripts/preflight.sh
<pass/fail/skip summary>

KEEL_MILESTONE=M7 scripts/preflight.sh
<pass/fail/skip summary>

scripts/m8-benchmark.sh --mode full --enforce
<metric table>
```

If SQL validation uses network access for Go module resolution, record that
fact. If it uses a prewarmed module cache, record the cache provenance.
